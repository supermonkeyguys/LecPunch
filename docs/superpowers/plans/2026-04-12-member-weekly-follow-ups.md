# 成员列表与周历史后续计划（2026-04-12）

## 背景

本轮已完成：

- 成员列表增加 `成员年级`
- 成员列表支持 `年级筛选`
- 成员列表支持按 `累计打卡时长区间` 筛选
- 成员列表支持按 `打卡时长 / 打卡次数` 排序
- 周历史补充 `周标识 / 周范围`
- 周历史进度条改为基于 `weeklyGoalSeconds` 计算

当前实现已经可用，但下面几个点仍建议进入下一轮计划。

## 建议项

### 1. 明确“打卡时间段”的产品语义

当前成员列表里的“时间段”实现为 `累计打卡时长范围（小时）`。

需要确认后续是否要扩展为：

- `累计时长范围`，保持现状
- `某个日期范围内的累计时长`
- `某天某时段的打卡记录筛选`

如果产品想要的是后两者，就不应该继续只靠前端表格筛选，而应新增服务端查询参数和聚合口径。

### 2. 统一团队榜与成员列表的可见范围

当前存在两个不同口径：

- Dashboard 团队榜默认是“同年级成员”
- Members 页面是“全团队成员”

这两者都各自合理，但容易让用户误以为数据不一致。

建议下一轮明确：

- 是否让 Members 页面也支持“同年级 / 全团队”切换
- 是否在 Dashboard 团队榜标题里直接写明“同年级”
- 是否把两处筛选条件同步到统一的查询模型

### 3. 周历史需要更强的状态说明

当前周历史已能显示周范围和目标进度，但仍缺少更明确的解释性信息：

- 当前周目标是多少，为什么是这个值
- 本周和历史周是否都按当前年级目标计算
- 作废 session 是否已计入或被排除

建议补一个轻量说明区，直接解释统计口径，减少用户误读。

### 4. 历史目标值需要考虑“按周快照”

现在周历史的进度分母来自当前登录用户的 `weeklyGoalSeconds`。

这在当前业务下可接受，但如果将来出现以下情况就会失真：

- 年级规则变化
- 用户年级跨学期变化
- 历史周目标规则调整

后续如果要保证历史周进度绝对准确，建议服务端返回“每周目标值”而不是只返回当前目标值。

### 5. 成员列表筛选条件建议同步到 URL

当前成员列表的筛选和排序只存在于页面内状态。

带来的问题：

- 刷新后条件丢失
- 返回列表页后上下文丢失
- 无法分享“某个筛选结果”的链接

建议下一轮把以下条件同步到 URL：

- `enrollYear`
- `minHours`
- `maxHours`
- `sort`
- `search`

## 推荐顺序

1. 先确认“打卡时间段”的产品语义
2. 再统一 Dashboard 与 Members 的团队口径
3. 然后补周历史统计口径说明
4. 最后决定是否升级为 URL 状态和服务端增强查询
