# 成员列表与周历史后续计划（2026-04-12）

## 背景

本轮已完成：

- 成员列表增加 `成员年级`
- 成员列表支持 `年级筛选`
- 成员列表支持按 `累计打卡时长区间` 筛选
- 成员列表支持按 `打卡时长 / 打卡次数` 排序
- 周历史补充 `周标识 / 周范围`
- 周历史进度条改为基于 `weeklyGoalSeconds` 计算

当前跟进项里，代码侧已基本收束，剩余一个产品语义待确认项。

## 已完成

### 1. 统一团队榜与成员列表的可见范围

已完成以下调整：

- Dashboard 团队榜明确按“同年级成员”展示
- Members 页面支持“全团队 / 同年级”范围切换
- Dashboard 跳转成员列表时可带入对应范围

### 2. 周历史补充统计口径说明

已补充轻量说明区，用于解释周历史的核心口径：

- 周范围按周标识展示
- 进度条按周目标值计算
- 作废 session 仍计入次数，但时长按 `0` 处理

### 3. 成员列表筛选条件同步到 URL

已将以下条件同步到 URL，便于刷新保留和结果分享：

- `scope`
- `enrollYear`
- `minHours`
- `maxHours`
- `sort`
- `search`

### 4. 历史周目标按周快照返回

已完成按周目标快照支持：

- 新产生的 session 会记录 `weeklyGoalSecondsSnapshot`
- 周历史优先按每周快照计算进度
- 旧数据若无快照，当前回退到用户当前年级目标

如果后续要求“规则变更后的历史周仍绝对准确”，仍需要额外的数据迁移或补录方案。

## 待确认

### 1. 明确“打卡时间段”的产品语义

当前成员列表里的“时间段”实现为 `累计打卡时长范围（小时）`。

仍需确认后续是否要扩展为：

- `累计时长范围`，保持现状
- `某个日期范围内的累计时长`
- `某天某时段的打卡记录筛选`

如果产品想要的是后两者，就不应该继续只靠前端表格筛选，而应新增服务端查询参数和聚合口径。

## 推荐顺序

1. 先确认“打卡时间段”的产品语义
2. 若确认需要日期/时段维度，再补服务端查询参数与聚合口径
3. 最后再决定是否需要继续扩展成员列表筛选 UI
